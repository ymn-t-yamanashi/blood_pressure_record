# blood_pressure_recordプロジェクト Docker 対応 仕様（詳細）

## 1. 目的
- Phoenix アプリを Docker で起動できるようにする
- DB はコンテナ内に同梱せず、ホスト上の既存 DB をそのまま利用する
- AI 処理はコンテナに入れず、ホスト上のプロセスとして実行する

## 2. スコープ
- 対象
  - Web アプリケーション実行環境のコンテナ化（Phoenix）
  - Docker 起動用設定（`Dockerfile`、`docker-compose.yml`、環境変数定義）
  - コンテナからホスト DB/AI サービスへの接続
- 非対象
  - DB サーバー自体の Docker 化
  - AI 実行基盤の Docker 化
  - 既存 DB データ移行

## 3. 全体構成
- `web` コンテナ
  - Phoenix アプリのみを実行する
- `db`（ホスト）
  - 現在運用中の DB を利用する
  - DB ファイル配置・インスタンス配置は現行維持（移動しない）
- `ai`（ホスト）
  - AI 処理プロセスをホストで起動し、Web から HTTP で呼び出す

## 4. DB 要件（コンテナ外配置）
- DB はホスト側で常駐すること
- DB 接続先は環境変数で切り替えること
  - 例: `DB_HOST`, `DB_PORT`, `DB_USERNAME`, `DB_PASSWORD`, `DB_NAME`
- 開発環境では、コンテナからホストへ到達可能なホスト名を使うこと
  - 例: `host.docker.internal`（Linux では compose 側の補助設定を利用）
- 既存 DB 配置を変えないこと
  - 既存のバックアップ・監視・運用手順を踏襲する
- DB 起動順制御は Docker 側では行わないこと
  - 起動前提: ホスト DB が利用可能であること

## 5. AI 実行要件（ホスト実行）
- AI 処理はホスト上で実行すること
  - 例: Python サービスや推論サーバーをホストで起動
- Phoenix コンテナは AI サービスを HTTP API 経由で呼び出すこと
- AI 接続設定は環境変数で管理すること
  - 例: `AI_BASE_URL`, `AI_TIMEOUT_MS`
- HTTP クライアントは `Req` を使用すること（`HTTPoison`/`Tesla`/`:httpc` は使用しない）
- 障害時要件
  - タイムアウト時は適切なエラーメッセージを返す
  - AI 側障害時はアプリ全体を停止させず、機能単位で失敗を扱う

## 6. Docker/Compose 要件
- `Dockerfile`
  - Phoenix アプリ実行に必要な依存のみ含める
  - 開発用と本番用で必要ならマルチステージ化する
- `docker-compose.yml`
  - 原則 `web` サービスのみ定義（DB サービス定義は作らない）
  - `environment` に DB/AI 接続情報を設定可能にする
  - 開発時はソースマウントとポート公開を行う
    - 例: `4100:4000`（ホスト 4100 を公開）
- `.env` または同等の手段で接続情報を外出しすること
- 機密情報（DB パスワード等）はリポジトリに直書きしないこと

## 7. アプリ設定要件
- `config/dev.exs` / `config/runtime.exs` で環境変数から接続情報を読み込む
- ローカル実行（非 Docker）と Docker 実行で同じ環境変数名を使う
- AI 連携先 URL は固定値にせず設定可能にする

## 8. 動作確認項目
- Docker 起動確認
  - `docker compose up` で Phoenix が起動する
- DB 接続確認
  - コンテナからホスト DB へ接続できる
  - 既存データの参照・登録・更新が可能
- AI 連携確認
  - コンテナからホスト AI API を呼び出せる
  - タイムアウト/エラー時のハンドリングが機能する
- 既存機能回帰確認
  - 血圧記録の主要画面・登録フローに回帰がない

## 9. 受け入れ条件
- Phoenix アプリが Docker コンテナで起動できる
- DB はコンテナ外（ホスト上の既存配置）を利用している
- AI 実行はホストで行い、Web から正常に連携できる
- 接続先は環境変数で切り替え可能
- README に起動手順と前提条件（DB/AI をホストで先に起動）が記載されている

## 10. リスクと対策
- リスク: コンテナからホスト接続できない
  - 対策: `host.docker.internal` と Linux 用 `extra_hosts` 設定を定義
- リスク: AI 応答遅延で画面体験が悪化
  - 対策: タイムアウト、リトライ方針、ユーザー向けメッセージを実装
- リスク: 環境差分による接続不整合
  - 対策: `.env.example` を整備し、必須変数を明文化

## 11. Ollama 設定要件（ホスト運用）
- Ollama はホストで常駐し、コンテナから到達可能なアドレスで待受すること
  - 必須: `0.0.0.0:11434` 待受
- `systemd` 利用時は `ollama.service` に次を設定すること
  - `Environment="OLLAMA_HOST=0.0.0.0:11434"`
- 反映手順
  - `sudo systemctl daemon-reload`
  - `sudo systemctl restart ollama`
  - `ss -ltnp | rg 11434` で `0.0.0.0:11434` を確認
- 利用モデルを事前取得すること
  - `ollama pull gemma3:27b`

## 12. .env 設定例（Docker Compose）
- `docker-compose.yml` の `OLLAMA_BASE_URL` は `.env` から読み込むこと
- `.env` 例
  - `OLLAMA_BASE_URL=http://host.docker.internal:11434/api`
- `host.docker.internal` で疎通できない環境では、コンテナ内から見たホスト IP を使うこと
  - 例: `OLLAMA_BASE_URL=http://172.17.0.1:11434/api`

## 13. Ollama 疎通確認手順
- コンテナ内から API 到達確認
  - `docker compose exec -T web curl -sS http://host.docker.internal:11434/api/tags`
- 到達できない場合の確認
  - Ollama 待受アドレス（`0.0.0.0` か）
  - `OLLAMA_BASE_URL` の値
  - ホスト FW が Docker ブリッジからの `11434/tcp` を拒否していないか
